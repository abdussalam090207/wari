# WARI Project Structure (Enterprise Standard)

To maintain a scalable, maintainable, and highly modular codebase, the WARI project strictly follows a feature-driven (Domain-Driven Design) architecture within the Next.js App Router environment.

## Directory Layout

```text
wari/
├── docs/               # Project documentation (Audits, Architecture, Design System)
├── public/             # Static assets (Images, SVGs, Fonts)
└── src/
    ├── app/            # Next.js App Router (Pages, Layouts, Loading, Error states)
    ├── components/     # Global, reusable UI components (Dumb/Presentational)
    │   ├── ui/         # Base shadcn/ui primitives (Button, Card, Input)
    │   └── layout/     # Global layout components (Navbar, Footer, Sidebar)
    ├── features/       # Domain-driven feature modules (Smart components)
    ├── hooks/          # Global React hooks
    ├── lib/            # Utilities, API clients, and animation libraries
    ├── styles/         # Global CSS and Tailwind directives
    └── types/          # Global TypeScript interfaces and types
```

---

## Strict Usage Rules

### 1. `src/app/`
*   **Purpose:** Routing only.
*   **Rule:** Files here (`page.tsx`, `layout.tsx`) should be as thin as possible. They should primarily import and compose components from `src/features/` or `src/components/layout/`.
*   **Data Fetching:** Server-side data fetching happens here before passing data down to feature components.

### 2. `src/features/` (The Core Engine)
*   **Purpose:** Grouping code by business domain rather than technical role. This is the hallmark of an enterprise React application.
*   **Structure:** Each feature folder acts as its own mini-application.
    ```text
    src/features/catalog/
    ├── components/     # Components specific to the catalog (ProductGrid, FilterBar)
    ├── hooks/          # Hooks specific to catalog (useFilter, useCart)
    ├── types/          # Types specific to catalog (Product, Category)
    └── index.ts        # Public API for this feature (Barrels)
    ```
*   **Rule:** A feature can import from `src/components/ui` or `src/lib`, but it should **never** import from another feature's internal folders. It can only import what is exposed in another feature's `index.ts`.

### 3. `src/components/`
*   **Purpose:** Truly global, highly reusable presentational components.
*   **Rule:** Components here must not have domain-specific business logic. A `Button` should not know about a "Wayang Product". It just receives an `onClick` handler and `children`.
*   **Sub-folders:**
    *   `/ui`: Reserved for base primitives (usually generated by shadcn/ui).
    *   `/layout`: For structural components like `Navbar`, `Footer`, `PageContainer`.

### 4. `src/lib/`
*   **Purpose:** Pure JavaScript/TypeScript functions, configurations, and third-party wrappers.
*   **Examples:** `utils.ts` (Tailwind class merging), `animations/` (Framer motion variants), formatters (currency, dates).
*   **Rule:** Files here should rarely contain JSX (except for specific animation wrapper components that act purely as utility logic).

### 5. `src/hooks/`
*   **Purpose:** Reusable React hooks that apply globally across multiple features.
*   **Examples:** `useMediaQuery`, `useScrollPosition`, `useLocalStorage`.
*   **Rule:** If a hook is only used by the Catalog feature, it belongs in `src/features/catalog/hooks/`, not here.

### 6. `src/types/`
*   **Purpose:** Global TypeScript declarations.
*   **Examples:** API response wrappers, common shared data models.
*   **Rule:** Like hooks, if a type is only used within a specific feature, define it within that feature's directory.

---

## Why This Architecture?
*   **Scalability:** As WARI grows (e.g., adding an admin dashboard or a blog), new features can be added without bloating the global components folder.
*   **Maintainability:** Finding code is easier. If there's a bug in the catalog filtering, developers know exactly where to look (`features/catalog`).
*   **Collaboration:** Multiple developers can work on different features simultaneously with minimal merge conflicts.
